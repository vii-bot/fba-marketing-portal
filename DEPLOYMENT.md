# Deployment Guide — Going Live

_Internal reference for launching the FBA Marketing Portal to production.
This is a repo file only — it is not part of the website. See
[SETUP.md](SETUP.md) for local development setup._

This guide is written for the current project setup: **Next.js 15 (App
Router)**, **Prisma + PostgreSQL** (local dev targets [Neon](https://neon.tech)),
and **Better Auth** for login/sessions. The recommended host is **Vercel**,
since that's what Next.js is built for and it requires the least
configuration.

Work through the sections in order — each one builds on the last.

---

## 1. Pre-Launch Checklist

Before deploying anywhere, make sure each of these is true.

### 1.1 Environment variables

These are the variables the app reads from `process.env`. Copy
[.env.example](.env.example) as a starting point, but note it's missing two
variables (`BETTER_AUTH_URL` and `ANTHROPIC_API_KEY`) that production needs —
add those too.

| Variable | Required? | Notes |
|---|---|---|
| `DATABASE_URL` | **Yes** | PostgreSQL connection string. **Use a separate production database from your local/dev one.** |
| `BETTER_AUTH_SECRET` | **Yes** | Random 32+ character secret used to sign sessions. Generate a fresh one for production with `openssl rand -hex 32` — do not reuse your local dev secret. |
| `NEXT_PUBLIC_APP_URL` | **Yes** | The production URL, e.g. `https://portal.fatbearagency.com`. Used by `next.config.ts` to allow server actions from your real domain, and by `lib/auth.ts` for Better Auth's trusted origins. |
| `BETTER_AUTH_URL` | **Yes** | Same production URL. Used by [app/api/invite/route.ts](app/api/invite/route.ts) to build the invite link an admin sends to a new hire (`{BETTER_AUTH_URL}/accept-invite?token=...`). If this is missing, invite links will point at `localhost`. |
| `ANTHROPIC_API_KEY` | Optional | Powers the SOP Builder's "AI Generate" draft feature ([lib/ai/generate-draft.ts](lib/ai/generate-draft.ts)). If unset, that feature falls back to a built-in mock generator — nothing breaks, the feature just produces a simpler placeholder draft. |
| `GOOGLE_CLIENT_ID` / `GOOGLE_CLIENT_SECRET` | Optional | Enables "Sign in with Google" on `/login` via Better Auth ([lib/auth.ts](lib/auth.ts)). OAuth 2.0 Web application credentials from Google Cloud Console, with redirect URI `{BETTER_AUTH_URL}/api/auth/callback/google`. If unset, the Google button on `/login` will error when clicked but email/password sign-in still works. |

### 1.2 Database connection

Confirm `DATABASE_URL` points at a real, reachable PostgreSQL database
(Neon, Supabase, Railway, etc. all work — Prisma's `provider = "postgresql"`).
Test it locally before deploying:

```bash
DATABASE_URL="<production-connection-string>" npx prisma db pull
```

If this command can read your schema without erroring, the connection string
is good.

### 1.3 Auth / session config

`lib/auth.ts` configures Better Auth with a `trustedOrigins` list — origins
that are allowed to make auth requests (sign in/up, session checks). It
already includes `NEXT_PUBLIC_APP_URL` automatically, so as long as that
environment variable is set to your real production URL, auth requests from
production will be accepted. No code change needed — just make sure the env
var is set correctly in your hosting provider.

### 1.4 Invite links

Invite links are generated by an admin from **Admin → Employee Database →
Add Employee** (the API route is `/api/invite`). The link is built from
`BETTER_AUTH_URL`, so once that env var is set to your production domain, a
newly-generated invite link should look like
`https://portal.fatbearagency.com/accept-invite?token=...` — not
`http://localhost:3000/...`.

### 1.5 Email sending

**There is currently no automated email sending** (no SMTP/Resend/SendGrid
integration). Invite links are generated in the admin UI and must be copied
and shared manually (e.g. via Slack, email, or whatever channel HR normally
uses to reach a new hire). This is fine for launch — just make sure whoever
is onboarding people knows to copy/send the link themselves.

### 1.6 Role / permission checks

All access control goes through [lib/permissions.ts](lib/permissions.ts)
(`isAdmin`, `isExecutive`, `isDepartmentManager`, `canManage*`/`canView*`
helpers). These don't depend on any environment configuration — they just
read `role`/`department` off the logged-in user, which come from the `User`
table. Nothing to configure here, but see §3 for how to test them.

### 1.7 Admin account exists

A brand-new production database has **no users at all**. You'll create the
first admin manually after the first deploy — see §2.3 and §4.5.

### 1.8 Migrations / schema

This project uses `prisma db push` (not `prisma migrate dev`/`deploy`) — there
is no `prisma/migrations` folder. That's fine for a project of this size, but
it means **schema changes are applied directly**, not via versioned migration
files. See §2.2 for how to apply the schema to production.

### 1.9 No test-only data exposed

- The **Danger Zone** (`/admin/danger-zone`, backed by `/api/admin/clear`)
  contains irreversible "delete all records of type X" actions, gated to
  `role === "admin"`. It's meant for clearing out test data during
  development — be careful who has the admin role in production, since
  they'll have access to this page.
- The seeded **Internal Documentation** pages (`npm run seed-docs`) are real
  reference content describing how the app works — they're meant to ship to
  production, not test data. No action needed.

## 2. Database Setup

### 2.1 Use a separate production database

Create a new Postgres database for production (a new Neon project/branch is
the easiest if you're already using Neon for dev). Do **not** point
production `DATABASE_URL` at your local/dev database — they should be
completely separate so local experiments never touch real data.

### 2.2 Apply the schema

With `DATABASE_URL` set to your **production** connection string (e.g. in a
one-off terminal session, or by running this from your local machine with the
env var temporarily overridden), run:

```bash
npx prisma generate
npx prisma db push
```

`db push` reads `prisma/schema.prisma` and makes the production database
match it — creating all tables (`User`, `Employee`, `SOP`, `Task`,
`DocPage`, etc.) on a fresh database. Run this once before your first deploy,
and again any time `prisma/schema.prisma` changes.

> **Be careful running `db push` against production after launch** — if a
> schema change would drop a column/table that has data in it, `db push` will
> warn you before applying. Read those warnings; don't blindly accept data
> loss on a production database. Take a backup first (see §7).

### 2.3 Seed data

Two seed scripts are available — run these against production after `db push`:

```bash
# Promote a user to admin (the user must have already signed up once)
DATABASE_URL="<production-url>" npm run make-admin -- your@email.com

# Populate the Internal Documentation CMS with starter reference pages
DATABASE_URL="<production-url>" npm run seed-docs
```

There's no separate "seed departments/roles" step — departments
(`DEPARTMENTS`/`ACTIVE_DEPARTMENTS`) and roles (`EXECUTIVE_ROLES`,
`MANAGER_ROLES`, etc.) are constants in [lib/utils.ts](lib/utils.ts), not
database rows, so they exist automatically.

### 2.4 Avoiding the dev database in production

The simplest safeguard: **never put your production `DATABASE_URL` in a local
`.env` file**. Only set it in your hosting provider's environment variable
settings (§4.2), and only export it temporarily in your terminal when you need
to run one of the commands above. `.env` is git-ignored, but it's still easy
to accidentally run a local `npm run dev` against the wrong database if both
URLs live in the same file — keep them physically separate.

## 3. Build and Test (Locally, Before Deploying)

### 3.1 Install dependencies

```bash
npm install
```

### 3.2 Run the app locally

```bash
npm run dev
```

Visit `http://localhost:3000` and confirm the app loads.

### 3.3 Lint and build checks

```bash
npm run lint
npm run build
```

`npm run build` runs `prisma generate && next build` — it will fail loudly if
there's a TypeScript error or a route that can't compile. This should
complete with no errors before you deploy.

### 3.4 Functional smoke test (do this against your dev database first)

Walk through each of these once locally — they cover the core flows that
must work before real employees touch the site:

- **Login** — log in at `/login` with an existing test account.
- **Invite flow** — as an admin, go to **Admin → Employee Database → Add
  Employee**, generate an invite link, open it in an incognito window, set a
  password, and confirm you land on `/setup` and then `/dashboard`.
- **Admin permissions** — confirm an `admin`/`Marketing Executive` account can
  see the full **Admin** section in the sidebar (Employee Database, LMS,
  Employee Requests, Taskboard, Contractor Requests, Creator Reports,
  Internal Documentation, etc.).
- **Employee permissions** — confirm a regular employee account does **not**
  see the Admin section, and only sees their own data on `/dashboard`,
  `/my-requests`, `/my-creators`, etc.
- **Department-based access** — log in as a Department Head/Manager/Account
  Manager and confirm **Admin → Employee Database** only shows that
  department's employees (not every department).
- **Submitting requests** — submit an Employee Request (COE) from
  `/requests`, an Attendance request (OT/Weekend OT/Leave) from
  `/attendance`, and a Contractor Change Request from
  `/admin/contractor-requests/submit`. Confirm each appears in its admin
  pipeline.
- **Exports** — from **Admin → Creator Reports**, run the CSV export
  (`/api/creator-reports/export`) and confirm a file downloads. From **Admin
  → Taskboard**, confirm the `.txt` export works.
- **SOP deadlines** — open **Resource Portal**, confirm a Published SOP with
  a deadline shows correctly, and that an exempted employee (via
  `SOPExemption`) doesn't get flagged as overdue.
- **Taskboard time tracking** — log a task on `/tasks` and confirm it shows
  up on the admin efficiency view at `/admin/tasks`.

## 4. Deploy to Vercel

1. **Push your code to GitHub** (if it isn't already) and **connect the repo
   to a new Vercel project** (vercel.com → "Add New… → Project → Import Git
   Repository").
2. **Add environment variables** — in the Vercel project, go to **Settings →
   Environment Variables** and add all five from §1.1 (`DATABASE_URL`,
   `BETTER_AUTH_SECRET`, `NEXT_PUBLIC_APP_URL`, `BETTER_AUTH_URL`, and
   optionally `ANTHROPIC_API_KEY`) for the **Production** environment.
   - For `NEXT_PUBLIC_APP_URL` and `BETTER_AUTH_URL`, you can initially use the
     `*.vercel.app` URL Vercel assigns you, then update both once you've set
     up a custom domain (§5) — just remember to redeploy after changing them.
3. **Set the production branch** — Vercel defaults to `main`/`master`; make
   sure this matches the branch you want to deploy from (Settings → Git).
4. **Deploy** — Vercel will automatically build using `npm run build`
   (`prisma generate && next build`), which is already the correct build
   command — no changes needed in Vercel's build settings.
5. **Check deployment logs** — in the Vercel dashboard, open the deployment
   and check the build log for errors. A successful build ends with the route
   summary (the same one `next build` prints locally).
6. **Confirm build success** — the deployment should show a green "Ready"
   status.
7. **Open the production URL** — visit the `*.vercel.app` URL Vercel gives
   you. You should see the `/login` page (an unauthenticated visit to `/`
   redirects there).
8. **Apply the database schema** — if you haven't already done §2.2/§2.3
   against this production database, do it now (with `DATABASE_URL` pointed
   at the production database, run `npx prisma db push` and the seed
   scripts).
9. **Create the first admin** — see §4.5 below.

### 4.5 Bootstrapping the first admin

A fresh production database has zero users. To get started:

1. Visit your production URL and sign up for an account through the normal
   `/login` "create account" flow (this creates a `User` row with the default
   `employee` role — there's no `Employee` HR record yet, which is fine).
2. Promote that account to admin directly against the production database:
   ```bash
   DATABASE_URL="<production-url>" npm run make-admin -- your@email.com
   ```
3. Log out and back in (role changes are picked up on next login).
4. As that admin, go to **Admin → Employee Database → Add Employee** to start
   onboarding everyone else through the normal invite flow — this keeps
   `Employee` and `User` records linked and role/department synced
   automatically via `lib/auth.ts`'s `databaseHooks.user.create.after`.

## 5. Domain Setup

1. **Add your domain in Vercel** — Project → Settings → Domains → enter your
   domain (e.g. `portal.fatbearagency.com`).
2. **Update DNS records** — Vercel will show you the exact record to add
   (usually a `CNAME` pointing at `cname.vercel-dns.com` for a subdomain, or
   an `A` record for an apex domain). Add this in your DNS provider's
   dashboard (wherever `fatbearagency.com`'s DNS is managed).
3. **Wait for DNS propagation** — this can take anywhere from a few minutes to
   a few hours. Vercel's Domains page will show a status check that turns
   green once it can see the DNS change.
4. **Confirm SSL/HTTPS is active** — Vercel automatically provisions a TLS
   certificate once DNS resolves. Visit `https://your-domain` and confirm the
   browser shows a valid lock/secure icon — no manual certificate setup is
   needed.
5. **Update environment variables** — once the custom domain is live, update
   `NEXT_PUBLIC_APP_URL` and `BETTER_AUTH_URL` in Vercel to the custom domain
   (e.g. `https://portal.fatbearagency.com`) and **redeploy** so the new
   values take effect (invite links and auth trusted origins both depend on
   these).

## 6. Post-Launch Testing

Once the site is live on its real domain, walk through this checklist with
the production database (ideally with one real admin account and one or two
test employee accounts you can clean up afterward):

- [ ] **Login** — `/login` works with email/password.
- [ ] **Invite acceptance** — generate an invite, open the link (now pointing
      at the real domain), set a password, and land on `/setup`.
- [ ] **Profile setup** — complete the `/setup` flow and confirm `/profile`
      shows the right name/role/department.
- [ ] **Admin dashboard** — `/admin` loads with stats and the full Admin nav.
- [ ] **Employee dashboard** — `/dashboard` loads for a regular employee with
      only the employee-facing nav.
- [ ] **Department Head access** — a Department Head/Manager only sees their
      own department's data in `/admin/employees`.
- [ ] **Employee Database** — add/edit an employee, confirm the shift
      schedule editor works.
- [ ] **SOP/Course deadlines** — Resource Portal shows deadline status
      correctly, and `SOPExemption` exclusions are respected.
- [ ] **Taskboard** — log a task as an employee, confirm it appears in
      `/admin/tasks`.
- [ ] **COE requests** — submit from `/requests`, confirm it appears in
      `/admin/requests` and can move through Pending → Processing →
      Completed/Rejected.
- [ ] **Contractor Change Requests** — submit from
      `/admin/contractor-requests/submit`, confirm it appears on
      `/admin/contractor-requests`.
- [ ] **My Schedule timezone display** — `/training/team-schedule` shows shift
      start times converted to the viewer's local timezone (not raw codes).
- [ ] **Attendance and Requests** — submit OT/Weekend OT/Leave from
      `/attendance`, confirm it's reflected in `/my-requests` and
      `/admin/attendance`.
- [ ] **My Creators** — `/my-creators` lists the creators assigned to the
      logged-in employee.
- [ ] **Creator Reports** — submit a report, confirm it's visible on
      `/admin/creator-reports` and moves through its status pipeline.
- [ ] **Team Notes** — add a note to a creator and confirm it shows in the
      creator's activity log.
- [ ] **Internal Documentation** — `/admin/internal-docs` loads for
      Admin/Executive, shows the 12 seeded pages across their categories, and
      a Department Head/Manager only sees pages whose `visibility` includes
      their role.
- [ ] **Export features** — Creator Reports CSV export and Taskboard `.txt`
      export both download successfully.
- [ ] **Audit logs** — perform a few actions (create an employee, submit a
      request, edit a doc page) and confirm they show up in the recent
      activity feed on `/dashboard`.

## 7. Production Safety Notes

- **Never commit `.env` or any file containing real secrets.** `.env` is
  already in `.gitignore` — double-check `git status` doesn't show it before
  any commit.
- **Don't expose secrets in client code.** Only variables prefixed
  `NEXT_PUBLIC_` are sent to the browser — `DATABASE_URL`,
  `BETTER_AUTH_SECRET`, `BETTER_AUTH_URL`, and `ANTHROPIC_API_KEY` must never
  get that prefix.
- **Use the production database only in production.** Keep production and
  local `DATABASE_URL` values in separate places (§2.4) so a local `npm run
  dev` session can never accidentally write to real employee data.
- **Keep admin accounts individual** — don't share one admin login across
  multiple people. Each admin should have their own account (via the normal
  invite flow with **Access → Super Admin** enabled) so audit logs
  (`AuditLog`, shown on `/dashboard`) reflect who actually did what.
- **Test permissions before giving the site to employees** — run through §3.4
  and §6 with real role/department combinations before announcing the
  rollout company-wide.
- **Back up the database before major changes** — before running `prisma db
  push` with a schema change, or using anything on **Admin → Danger Zone**,
  take a backup/snapshot of the production database (Neon supports
  point-in-time restore and branch-based backups).
- **Keep a rollback plan ready** — Vercel keeps previous deployments; if a
  deploy introduces a problem, you can instantly "Promote" a previous
  deployment back to production from the Vercel dashboard while you fix the
  issue.

## 8. Final Launch Checklist

- [ ] Production URL is working (`https://your-domain`).
- [ ] Admin login works.
- [ ] Employee invite works end-to-end (link → accept → setup → dashboard).
- [ ] Permissions work (admin vs. employee vs. department-scoped views all
      checked).
- [ ] Database is connected (production `DATABASE_URL`, schema pushed,
      `make-admin` and `seed-docs` run).
- [ ] Forms submit correctly (Employee Requests, Attendance, Contractor
      Change Requests, Creator Reports).
- [ ] Exports work (Creator Reports CSV, Taskboard `.txt`).
- [ ] Audit logs work (recent activity shows on `/dashboard`).
- [ ] Domain/SSL works (custom domain resolves over HTTPS).
- [ ] Ready to send to department heads for testing.
